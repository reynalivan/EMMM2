### Title

Core Collections & Safe Mode Boot Sequence Gaps

### Context

The UI lacked a boot-time lock mechanism for the Safe Mode PIN, ObjectList payloads for active mods were poorly typed, and Collections saving lacked physical disk validation and GC for dirty states.

### Changes

- Formatted `PinEntryModal` to optionally block all cancellation methods.
- Wired the PIN modal to block the `App.tsx` boot sequence for Unsafe mode when a PIN exists.
- Updated `ObjectSummary` to include a proper array of `active_mod_paths`.
- Parsed `active_mod_paths_joined` into `Vec<String>` within `src-tauri/src/services/objects/query.rs`.
- Added physical disk `exists()` validation to `create_collection`.
- Added `mod_count >= 1` invariant check to `create_collection`.
- Implemented explicit deletion of `kind = 'unsaved'` collections generated by the dirty state tracker after saving a new permanent collection.

### Impacted Files

- `src/App.tsx` (modified)
- `src/features/safe-mode/PinEntryModal.tsx` (modified)
- `src/types/object.ts` (modified)
- `src-tauri/src/database/object_repo.rs` (modified)
- `src-tauri/src/services/objects/query.rs` (modified)
- `src-tauri/src/services/collection_service.rs` (modified)
- `src-tauri/src/domain/errors.rs` (modified)

### Goal

Ensure strict PIN boundaries on application boot, display accurately typed active mods in ObjectLists, and prevent corrupted or conceptually empty Collections from being saved permanently.

### Impact

- Application boots are now cryptographically secure if a PIN is set and the previous session ended in Unsafe Mode.
- Collections with 0 active mods will be rejected by the backend.
- Invalid physical mod paths will be rejected during collection snapshots.
